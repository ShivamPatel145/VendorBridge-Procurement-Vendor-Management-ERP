01

Complete System Architecture

VendorBridge is a multi-tenant Procurement & Vendor Management ERP built on a decoupled frontend/backend architecture. The Next.js 15 frontend communicates exclusively with the Express REST API; all integrations (Resend, Cloudinary, Upstash, PDFKit) live in the backend service layer, never exposed to the browser.

Frontend

Next.js 15 App Router with SSR/ISR. TanStack Query for server state. React Hook Form + Zod for forms. Shadcn UI components.

Backend

Express.js REST API with layered middleware. Prisma ORM over Neon PostgreSQL. Upstash Redis for caching & sessions.

Infrastructure

Vercel for frontend edge delivery. Render for backend container hosting. Cloudinary for binary assets. Resend for transactional email.

Upstream Contexts

  • Identity Context owns User and Role — all other contexts reference User by FK only
  • Vendor Context is the supplier master — never managed from other contexts
  • Procurement Context initiates all RFQ-driven workflows

  • Approval Context reacts to Quotation selection events via domain events

  • Analytics Context reads all contexts but never writes back

Authentication Architecture

▸ JWT authentication flow with refresh token rotation
sequenceDiagram participant C as Client (Next.js) participant API as Express API participant REDIS as Redis participant DB as PostgreSQL C->>API: POST /auth/login {email, password} API->>DB: Find user by email DB-->>API: User record (hashed pw) API->>API: bcrypt.compare(password, hash) API->>API: Sign Access Token (15m) + Refresh Token (7d) API->>REDIS: Store refreshToken hash (key: userId) API-->>C: {accessToken, refreshToken} + HttpOnly cookie Note over C,API: Access token expires after 15 min C->>API: POST /auth/refresh {refreshToken in cookie} API->>REDIS: Validate refresh token hash REDIS-->>API: Valid API->>API: Rotate — issue new Access + Refresh tokens API->>REDIS: Invalidate old token, store new API-->>C: New {accessToken, refreshToken} Note over C,API: On logout C->>API: POST /auth/logout API->>REDIS: Delete refreshToken entry (blocklist accessToken) API-->>C: 200 OK

Access Token

JWT RS256 15-minute TTL. Payload: userId, role, orgId. Signed with RSA private key on backend.

Refresh Token

HttpOnly Cookie 7-day TTL. Stored as HMAC hash in Upstash Redis. Rotation on every use. Family invalidation on detect-reuse.

Password

bcrypt cost factor 12. Forgot-password flow: short-lived (10m) signed token via Resend. Token one-time-use stored in Redis.

Authorization Architecture (RBAC)

▸ Role-based access control matrix
graph LR subgraph Roles ADMIN[Admin] OFFICER[Procurement\nOfficer] MANAGER[Manager/\nApprover] VENDOR[Vendor] end subgraph Permissions P1[Manage Users] P2[Manage Vendors] P3[Create RFQ] P4[Submit Quotation] P5[Compare Quotations] P6[Approve/Reject] P7[Generate PO] P8[Generate Invoice] P9[View Reports] P10[View Activity Logs] end ADMIN --> P1 ADMIN --> P2 ADMIN --> P9 ADMIN --> P10 OFFICER --> P2 OFFICER --> P3 OFFICER --> P5 OFFICER --> P7 OFFICER --> P8 MANAGER --> P6 MANAGER --> P9 VENDOR --> P4

RBAC Implementation

Middleware reads role from the verified JWT payload and checks a static permission map. Resource-level ownership checks (e.g., vendor can only submit to RFQs they're assigned to) are enforced inside each service method via Prisma where clause scoping, not in middleware.

// rbac.middleware.ts
const PERMISSIONS: Record<Role, Permission[]> = {
  ADMIN:    ['manage:users','manage:vendors','view:reports','view:logs'],
  OFFICER:  ['manage:vendors','create:rfq','compare:quotations','generate:po','generate:invoice'],
  MANAGER:  ['approve:workflow','view:reports'],
  VENDOR:   ['submit:quotation','view:rfq','view:po'],
}

export const requirePermission = (permission: Permission) =>
  (req: Request, res: Response, next: NextFunction) => {
    const role = req.user.role
    if (!PERMISSIONS[role]?.includes(permission))
      return res.status(403).json({ error: 'Forbidden' })
    next()
  }

Design Principles

  • All PKs are UUIDs (v4) — no auto-increment integers exposed to clients
  • Soft-delete pattern (deletedAt) on Vendor and RFQ; hard-delete elsewhere
  • ActivityLog is append-only — no UPDATE or DELETE allowed at DB level via trigger
  • GSTIN stored as text with format validation at application layer
  • Money values: Decimal(12,2) — never Float

Indexing Strategy

  • users.email — unique index
  • vendors.gstin — unique index
  • rfqs.status, createdAt — composite index for dashboard queries
  • quotations.rfqId, vendorId — composite unique
  • activity_logs.userId, createdAt — for audit trail pagination
  • invoices.status, dueDate — for overdue detection

Security Architecture

Transport & Headers

  • TLS 1.3 enforced end-to-end (Vercel + Render auto-provision)
  • Helmet.js sets CSP, HSTS, X-Frame-Options, X-Content-Type-Options
  • CORS: whitelist vendorbridge.app origin only
  • Refresh token in HttpOnly; Secure; SameSite=Strict cookie
  • Access token in memory (never localStorage)

Input Validation

  • All API inputs validated with Zod schemas before business logic
  • Prisma parameterized queries — no raw SQL (SQL injection impossible)
  • File upload: MIME type + magic-bytes validation (not just extension)
  • Max upload size enforced at Multer + Nginx level
  • GSTIN format validated with regex at schema layer

Rate Limiting

  • express-rate-limit backed by Upstash Redis
  • Auth endpoints: 5 req/15min per IP (brute-force protection)
  • General API: 100 req/min per authenticated user
  • PDF generation: 10 req/min (CPU-intensive endpoint)

Data Protection

  • ActivityLog is immutable — DB trigger blocks UPDATE/DELETE
  • Sensitive env vars in Vercel/Render secrets — never in codebase
  • Cloudinary signed URLs for all private documents (not public)
  • PII fields (email, phone) indexed — never logged in plaintext
  • No vendor can see another vendor's quotation (enforced in service layer)

Key Architecture Decisions Summary

  • Monorepo (Turborepo) — shared types between Next.js and Express eliminate API contract drift
  • Modular Monolith (not microservices) — right size for a hackathon MVP; clean boundaries enable future decomposition
  • Neon Serverless PostgreSQL — scales to zero for dev/staging; branching per PR for database migrations preview
  • Upstash Redis — HTTP-based, works from edge/serverless without persistent TCP connections
  • Prisma ORM — type-safe queries, migration management, and Neon adapter built-in
  • JWT RS256 asymmetric signing — public key can be distributed to edge functions for token verification without calling the API
  • ActivityLog immutability via DB trigger — compliance and audit trail integrity at the database level, not just application level
  • Cloudinary signed URLs — vendor-sensitive documents never served with a permanent public URL
